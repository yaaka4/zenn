# はじめに

保育士向け転職サイトとホテル向け転職サイトの開発組織にて、エンジニアリングマネージャー（EM）を担当しているfurusawaです。以前はスクラムマスターとしてチームの現場に携わっていましたが、現在はEMとして開発組織全体のチーム設計や運営に関わっています。

今期、SEO施策に専念するアジャイルチームを新設しました。この記事では、なぜそのチームが必要だったのか、どのようにチーム体制を設計・合意形成したのか、そして実際に動かしてみてわかったことを紹介します。

EMとして「事業の課題をどうチーム設計で解くか」を考えるプロセスをお伝えできれば幸いです。

# 背景：なぜSEOチームが必要だったのか

## SEOコンサルの導入と、リソース確保の要望

今期から外部のSEOコンサルに入ってもらうことになりました。コンサルから提案される施策を着実に実行するには、SEO専用のエンジニアリソースが必要だという事業側からの要望がありました。

## 既存チームでSEO施策が後回しになる構造的問題

それまでの運用では、通常のプロダクト施策を担うスクラムチームの中で、他の施策と並べて優先順位を決め、SEO施策に取り組んでいました。

しかしこの運用には、構造的な問題がありました。

**SEO施策は効果インパクトを説明しにくい。**

グロース施策であれば「このボタンのCVRが◯%改善される見込み」と数値で語りやすいのですが、SEO施策は「検索順位が上がるかもしれない」という不確実性が高く、優先順位をチーム内で上げにくい状況でした。結果として、軽微な改善でも後回しにされ続けていました。

# チーム体制の検討と合意形成

## 2つの案を比較検討した

事業要望を受けて、EMである私はいくつかの案を検討しました。主な候補は以下の2つです。

| 案 | 概要 | メリット | デメリット |
|----|------|----------|------------|
| **A案：SEOチームを切り出す** | SEO施策専用のチームを新設し、専用リソースを確保する | SEO施策が後回しにならない。週次サイクルで小さく速く動ける | チームが増えることによる管理コスト |
| **B案：既存チーム内で優先度を分離** | 既存スクラムチームの中で、通常施策とSEO施策を別軸で優先度付けする | チームを増やさずに済む | 依然としてリソース競合が発生しやすく、根本解決にならない |

SEO施策が後回しになるという根本的な構造を変えるには、リソースを物理的に分離するA案が適切だと判断しました。

## 上長・CTO・ステークホルダーとの合意形成

チーム体制の変更は、開発組織だけでなく事業サイドにも影響します。上長・管掌役員（CTO）・ステークホルダーに対して、課題の整理・比較検討の過程・判断の根拠を説明し、合意を形成した上で進めました。

「開発組織が事業課題に対して主体的に解を考え、動く」というプロセスを大切にしています。

# SEOアジャイルチームの構成

## 多国籍チームという特徴

このチームの一番の特徴は、**多国籍なメンバー構成**です。

| 役割 | 詳細 |
|------|------|
| 正社員エンジニア（チームリーダー） | 日本在住。チーム全体のとりまとめ役 |
| 業務委託エンジニア（ブリッジSE） | バングラデシュ出身、日本在住。開発を担いつつ、英語が堪能なためブリッジSEの役割も兼任 |
| 業務委託エンジニア（オフショア） | 台湾在住。日本語は話せないため、ブリッジSEを経由して英語でコミュニケーションを取る |
| プロダクトオーナー | 通常のプロダクト開発チームのPOも兼任 |
| SEO担当者（マーケ部門）| 2名。SEOコンサルからの施策提案を一次受けし、開発チームへ渡す役割 |

ブリッジSEはバングラデシュ出身ということもあり、オフショアエンジニアとの英語コミュニケーションを自然にカバーしてくれています。Slackでのテキストコミュニケーションではお互いにSlackの翻訳機能を活用し、言語の壁をスムーズに乗り越えています。

## プロダクト横断チームという設計

もう一つの特徴は、**プロダクト横断でSEO施策を担う**点です。

従来は「プロダクト単位」で開発チームが存在していましたが、SEOチームは保育士向け転職サイトとホテル向け転職サイトの両プロダクトを対象にしています。SEO施策には「あるプロダクトで効果のあった施策を別プロダクトに横展開できる」という特性があるため、横断チームとして設計することで効率よく改善を進められます。

# 開発プロセス：小さく・速く・週次で回す

チームの人数が少なく、施策の粒度も小さいため、スクラムの形式は取らず、軽量なアジャイルプロセスで運用しています。スクラムのセレモニーをイメージした会議体は以下の通りです。

| セレモニー（イメージ） | 実際の会議体 | 参加者 |
|-----------------------|------------|--------|
| リファインメント | 週次SEO定例 | 自社メンバー全員（PO・SEO担当・エンジニア） |
| デイリースクラム | 朝会（日本メンバー） | エンジニア2名 |
| デイリースクラム | 朝会（英語、事前実施） | ブリッジSE・オフショアエンジニア |
| プランニング＋レトロ | 週次開発定例 | エンジニアのみ |

SEO施策は1つ1つの改修ボリュームが小さくなりやすいため、**週次サイクルで毎週小さくデプロイ**することができています。「小さく速く仮説検証を回す」というアジャイルの考え方と相性が良いです。

# やってみてわかったこと

## よかったこと

**後回し問題が解消された**

専用リソースを確保したことで、今まで後回しにしてきたSEO改善をスムーズにリリースできるようになりました。チームが独立したことで、他の施策との優先順位争いが発生しなくなったのが最大の効果です。

**プロダクト横断での横展開が機能している**

あるプロダクトで実施したSEO施策を別プロダクトへ横展開するサイクルが生まれ、改善の効率が上がっています。

**SEO成果も出始めている**

Googleのコアアップデートの影響もありますが、SEO順位が上昇傾向にあり、特定のキーワードで1位を獲得する回数が増えてきています。

## まだ残っている課題

**バケツリレーになりやすい**

関係者が多く、情報の流れが直線的になりがちな点が課題です。

```
SEOコンサル → マーケSEO担当者 → チームリーダー → ブリッジSE → オフショアエンジニア
```

コミュニケーションの層が多いと、情報の鮮度や意図が伝わりにくくなるリスクがあります。

現在の改善案として、**マーケのSEO担当者を朝会や週次開発定例に参加してもらう**ことを検討中です。開発チームと事業サイドの距離を縮めることで、施策の意図や優先度をリアルタイムに共有できる状態を目指しています。

# まとめ

SEOチームの新設を通じて、EMとして意識したのは **「事業課題をチーム設計で解く」** という考え方です。

「SEO施策が後回しになる」という問題の根本には、リソース競合という構造的な課題がありました。その課題に対して複数の解決案を検討し、上長・CTO・ステークホルダーと合意形成しながら主体的に進めることができました。

開発組織が事業にコミットするために、ただ言われた開発をこなすだけでなく、**「どういうチーム体制が事業にとって最善か」を自分たちで考え抜き、行動する**。そういった動き方を大切にしています。

弊社では、エンジニアリングマネージャーが事業課題に対して主体的に関わり、開発組織と事業が一体となってプロダクト成長を目指せる環境です。多国籍チームの運営や、横断的なチーム設計など、挑戦できる機会もたくさんあります。
